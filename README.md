# Transactions processor

## Description

This is an implementation of simple payment engine that reads a series of transactions from CSV, updates the client accounts, handles the disputes and chargeback and outputs the state of clients accounts as a CSV.

There is an assumption the transaction is a `u32` value.

## How to run

```sh
cargo run -- tests/input.csv > output.csv
```

## Testing

Most of the edge-cases is covered by unit tests in `Payments` module

```sh
cargo test
```


## Assumptions

### Transaction ID

| Decision | Motivation |
|----------| -----------|
| Any transaction with duplicated ID will be blocked  | The Instruction contains information that transaction IDs are globally unique.|

### Client ID

| Decision | Motivation |
|----------| -----------|
| The transaction generated by `client_id` "A" cannot be altered (Disputed etc..) by the client with id "B" | Transaction are attached to one account |

### Amount

| Decision | Motivation |
|----------| -----------|
| The transactions with precision greater than 4 are rejected | The instruction contains the information about precision. Additionally the engine should round values for the client|
| The negative (below or equal 0) 'amount' is rejected | Depositions of negatives shouldn't be possible |

### Locking

| Decision | Motivation |
|----------| -----------|
| all operations (Withdraw, Deposit, etc.) are forbidden when the lock is on | As the instruction wasn't clear, I assumed that. Lock suggests immutability of account |

### Dispute

| Decision | Motivation |
|----------| -----------|
| Transaction that has been previously resolved, can be put into `Dispute` state again |  As the instruction wasn't clear. I assumed that the customer my want to try to chargeback the money second time|
|It is impossible to put the `Withdraw` transaction into `Dispute` and consequently 'Chargeback' state. | Instruction says, that for Chargeback the total funds should **decrease**. When the `Withdraw` is reversed it should **increase**. Additionally I'm not sure if we can manage the money that left the account|

### Performance

| Decision | Motivation |
|----------| -----------|
| Marshaling and account processing are separated and put into different threads.| performance optimisation which could be useful in case of large volume of data|

### Extensibility

| Decision | Motivation |
|----------| -----------|
|The input (`io:Read`) is exposed to the root | we may want to attach the different data source |
|The output (`Sender<ClientID, Client)>`) is exposed to the root | we may want to use a different destination for data|
